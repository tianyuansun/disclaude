# 每日灵魂拷问

每天上午 10 点分析聊天/工作记录，生成灵魂拷问并发送到话题群。

## 配置说明

**重要**: 使用前请将 `chatId` 替换为实际的飞书 Chat ID（可以是话题群或普通群），并将 `enabled` 设为 `true`。

## 执行步骤

### 1. 获取聊天记录文件

```bash
ls workspace/chat/*.md 2>/dev/null || echo "No chat files found"
```

如果 `workspace/chat/` 目录不存在或为空，跳过本次执行。

### 2. 分析聊天记录

对于每个聊天记录文件：

1. 读取文件内容（重点关注最近 7 天的消息）：
   ```
   Read workspace/chat/{chatId}.md
   ```

2. 分析记录，识别以下内容：
   - **有趣的决策**: 技术选型、架构决策、流程变更等
   - **潜在改进点**: 重复出现的问题、效率瓶颈、技术债务
   - **值得反思的问题**: 未充分讨论的权衡、需要验证的假设

### 3. 生成灵魂拷问

基于分析结果，生成一个开放式的、引人思考的问题：

```markdown
## 🤔 今日灵魂拷问

分析最近的聊天/工作记录，发现一个有趣的问题：

「{核心问题}」

**背景**:
{问题背景说明}

**值得思考**:
1. {思考问题1}
2. {思考问题2}
3. {思考问题3}

欢迎在群里讨论 👇
```

### 4. 发送灵魂拷问

使用 `send_user_feedback` 将灵魂拷问发送到配置的 chatId：

```
send_user_feedback({
  chatId: "{配置的chatId}",
  message: "{生成的灵魂拷问内容}"
})
```

## 质量要求

### 好的灵魂拷问:
- ✅ 基于实际工作/决策
- ✅ 开放式（多种有效观点）
- ✅ 引人思考（让人停下来想一想）
- ✅ 与团队相关

### 避免:
- ❌ 是非题（是/否就能回答）
- ❌ 答案明显的问题
- ❌ 批评特定个人的问题
- ❌ 与实际工作无关的问题

## 错误处理

1. 如果聊天记录文件读取失败，跳过该文件继续处理其他文件
2. 如果分析后没有找到值得讨论的话题，不发送消息
3. 如果 `send_user_feedback` 失败，记录日志并重试一次

## 示例输出

```markdown
## 🤔 今日灵魂拷问

分析最近的聊天/工作记录，发现一个有趣的问题：

「在处理用户认证模块时，我们选择了 JWT 而非 Session。这个决策是否正确？」

**背景**:
- JWT 无状态，适合分布式部署
- Session 更安全，但需要共享存储
- 团队对安全性 vs 可扩展性有不同看法

**值得思考**:
1. 我们是否低估了 JWT 的安全风险？
2. "无状态优先"在什么情况下是正确的策略？
3. 如果用户量增长 10 倍，当前选择还能支撑吗？

欢迎在群里讨论 👇
```

## 与话题群的配合

当 Issue #721（话题群基础设施）合并后，可以将 `chatId` 设置为话题群，实现 BBS 模式的讨论：

- Agent 主动推送灵魂拷问
- 不预期用户一定有响应
- 开放式讨论，引发思考
